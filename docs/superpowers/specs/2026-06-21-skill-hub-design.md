# Skill Hub Desktop：本地 Skill 与 Plugin 管理器设计

## 目标

构建一个 Cockpit 风格的本地桌面应用，统一发现、查看、安装与删除本机 Codex 和 Claude Code（CC）的 skills 与 plugins。第一版不使用账号、云端服务或内置在线目录。

用户可以看见本地已安装资源、其真实文件路径和所属宿主；可从 GitHub 仓库或目录 URL 安装；并可将资源安装到另一个受支持宿主或移入系统废纸篓。

## 范围

### MVP 包含

- 本地桌面应用，采用 Tauri + React。
- Codex 与 Claude Code 的 skills 和 plugins 发现、列表、搜索与筛选。
- 资源详情：名称、类型、已安装宿主、发现路径、兼容状态、可用操作。
- 从 GitHub 仓库或目录 URL 安装；安装前预览资源类型、目标路径、兼容性和冲突。
- 将一个可兼容资源安装到另一宿主。
- 二次确认后移入系统废纸篓，不直接永久删除。
- 启动扫描与手动刷新；局部扫描错误以告警显示，不阻断其他结果。

### MVP 不包含

- 登录、云同步、团队目录或远程 registry。
- 内置搜索型在线 skill/plugin 市场。
- 自动覆盖同名资源。
- 读取、显示、复制或修改 `.env`、token、API key 和其他凭据。

## 架构

```text
React GUI
  ├─ Overview: 总数、每宿主可用数、扫描告警
  ├─ Skills / Plugins: 搜索、筛选、资源列表
  ├─ Detail drawer: 路径、兼容性、跨宿主安装、移入废纸篓
  └─ Sources: GitHub URL 安装和预览
          │ Tauri commands
          ▼
Rust service layer
  ├─ Codex adapter
  ├─ Claude Code adapter
  ├─ Git installer
  ├─ Conflict detector
  └─ Trash service
          ▼
Local filesystem and GitHub repositories
```

界面只依赖统一资源模型，不能解析或依赖外部 CLI 的文本输出。宿主目录、格式识别、目标路径和安装策略全部封装于独立适配器中。

## 统一领域模型

每个扫描结果至少包含：

- `id`：稳定的本地资源标识。
- `name`：显示名称。
- `kind`：`skill` 或 `plugin`。
- `hosts`：可用的宿主集合（Codex、Claude Code）。
- `locations`：每个发现目录的绝对路径与状态。
- `compatibility`：可直接安装、需转换、未知或不支持。
- `metadata`：只解析安全的清单/说明文件字段；不读取凭据文件。
- `warnings`：目录不可读、格式无法识别、名称冲突等非致命问题。

同名但路径不同的资源必须保留为独立记录，直到用户明确选择合并或移除；MVP 不做自动合并。

## 宿主适配器

`HostAdapter` 提供以下职责：

- 返回当前机器中该宿主的候选配置、skill 和 plugin 根目录。
- 在根目录内安全枚举资源，识别其类型和最低可用元数据。
- 判定资源是否能安装到该宿主，并返回明确原因。
- 给出安装目标路径与冲突结果。
- 执行受控的复制/写入，不越过已确认的宿主根目录。

Codex 与 Claude Code 各自实现适配器。检测规则和默认路径必须集中在适配器内，前端不得硬编码路径或资源格式。

## 核心操作

### 扫描

应用启动和用户点击刷新时并行扫描两个适配器。不存在或无权限的目录产生宿主级告警；该宿主其他可扫描目录及另一宿主的结果仍然显示。

### 从 GitHub URL 安装

1. 用户输入 GitHub 仓库 URL 或仓库内目录 URL，并选择目标宿主。
2. 后端克隆或获取到临时目录，且不写入目标目录。
3. 识别可安装的 skill/plugin，校验目标宿主兼容性和名称冲突。
4. 前端展示资源、最终绝对目标路径、冲突和将发生的操作。
5. 用户确认后，后端执行安装并刷新扫描结果。

格式未知、URL 无法访问或目标有同名资源时，安装中止并解释原因。MVP 默认拒绝覆盖；覆盖策略不隐式发生。

### 跨宿主安装

详情抽屉仅对适配器标为可兼容的资源显示“安装到 Codex / Claude Code”。执行前与 URL 安装相同：显示目标路径和冲突；确认后复制或由适配器转换。若需要未实现转换，按钮禁用并显示原因。

### 删除

用户在详情抽屉选择具体宿主和路径。确认对话框明确说明资源名称、宿主和绝对路径。后端调用系统废纸篓而非永久删除；失败时不进行回退删除，并把系统错误呈现给用户。

## GUI

左侧固定导航：`Overview`、`Skills`、`Plugins`、`Sources`、`Settings`。

- `Overview` 显示资源总数、按宿主拆分的数量和扫描告警。
- `Skills` 与 `Plugins` 共用资源表：搜索、宿主筛选、状态筛选、路径副标题与兼容标签。
- 资源行打开右侧详情抽屉，承载路径、元数据、兼容状态和操作。
- `Sources` 提供 GitHub URL 输入、目标宿主选择、安装预览与确认。
- `Settings` 仅显示当前检测到的宿主根目录和刷新入口；MVP 不支持自定义任意写入路径。

## 安全与错误处理

- 后端只在适配器允许的根目录和临时克隆目录操作文件。
- 不读取、展示、复制或修改 `.env`、token、key、credential 等敏感文件；扫描遇到它们时跳过。
- 一切写入、跨宿主安装和删除必须以最终绝对路径进行确认。
- 所有错误转化为可操作状态：扫描告警、安装预检错误、冲突、权限错误或废纸篓失败。
- 操作日志只保留资源名称、类型、宿主和路径；不得包含 URL 中的访问令牌或任何凭据。

## 验证策略

- Rust 单元测试：路径边界、目录扫描、资源识别、冲突检测、跨宿主安装判定、敏感文件跳过和废纸篓调用。
- React 组件测试：资源筛选、详情抽屉、安装预览、确认对话框和错误状态。
- 端到端测试：临时 Codex/Claude Code 根目录下的扫描、Git URL 安装、跨宿主安装、冲突拒绝和移入废纸篓流程。
- 手工验证：在实际本地 Codex 与 Claude Code 目录上确认只读取预期元数据，且 UI 中路径与磁盘一致。

## 交付标准

1. 未安装其中一个宿主时，应用仍可启动并清晰标示缺失状态。
2. 已安装的 resources 显示正确类型、宿主、路径和兼容状态。
3. 有效 GitHub URL 可在确认预览后安装到指定宿主。
4. 冲突、不可用 URL、权限不足和不支持格式不会导致部分写入。
5. 删除只会进入系统废纸篓，确认提示的路径与实际操作一致。
6. 敏感文件和内容不会显示或进入操作日志。
